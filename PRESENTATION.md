«Мы реализовали **цифрового двойника** станции сортировки Ozon: WMS по штрихкоду назначает **рукав** (направление), YOLO — **тип упаковки**; ПЛК рассчитывает момент отвода. Тот же код — на видео, RTSP и PyBullet. Спорные кейсы — **LLM-арбитр** с audit-логом. Подробно: [docs/BUSINESS_RULES.md](docs/BUSINESS_RULES.md).»

## Слайд 1 — Проблема

**Заголовок:** Сортировка на конвейере = узкое горлышко хаба

- Поток посылок растёт → нужна автоматизация без ручного разбора
- Ozon: классический CV на ОВХ не универсален → **нейросети сверху** ([кейс Ozon Tech](https://www.pvsm.ru/machine-learning/391187))
- Хакатон: обнаружить → классифицировать → отсортировать (железо **или** симуляция)

**Визуал:** фото NBS-сортера / схема ленты сверху.

## Слайд 2 — Наше решение (одна схема)

```
FIELD  →  WCS (CV + трекинг + ПЛК)  →  WMS  →  Актуатор
              │
              └─ ScanStation: блок коррекции маршрута
                    ① barcode  ② WMS  ③④ LLM-арбитр?
```

**Ключ:** разделение ответственности как на проде (см. [BUSINESS_RULES.md](docs/BUSINESS_RULES.md)):
- WMS — **КУДА** (рукав по штрихкоду / кластеру)
- ПЛК — **КОГДА** (`execute_at_frame`)
- CV — **ЧТО** (тип упаковки) + **ГДЕ** (трекинг ≈ энкодер)

## Слайд 5 — WMS без ML

`config/routes.yaml` — **бизнес-правила**, не веса нейросети:

```
Штрихкод 460…  →  chute_a  (кластер / направление)
Штрихкод 461…  →  chute_b  (другой кластер)
YOLO: box      →  тип упаковки (в логе class), не город

Один тип box может уехать в chute_a или chute_b — решает код, не форма.
no match       →  zone_reject
```

**Демо PyBullet:** на объектах нет EAN → временный fallback `by_class` (см. комментарии в yaml).

**Фраза:** «Нейросеть видит форму; WMS по штрихкоду знает, куда везти».

## Слайд 6 — Тайминг (ПЛК-логика)

```
SCAN LINE ──── Δx ──── ACTUATION LINE ──── зона
              ETA = Δx / belt_speed + lead_time
```

- `CommandQueue` — **без `time.sleep`** в цикле
- Несколько объектов на ленте одновременно

**Живое сообщение в консоль/лог:**  
`[scheduled] track #17 → chute_a через 67 кадров`

## Слайд 7 — LLM Arbitrator (блок коррекции маршрута)

**Место:** шаги ③–④ внутри `ScanStation` на SCAN LINE — см. [ARCHITECTURE.md](ARCHITECTURE.md) §2a.

**Не замена YOLO — корректор спорного маршрута:**

| Триггер | Действие |
|---------|----------|
| confidence < 0.55 | LLM смотрит crop + metadata |
| barcode ≠ CV class | арбитраж зоны |
| rate limit | fallback на WMS |

**Выход:** `arbitrator.jsonl` с reasoning — **объяснимый AI** для оператора.

**Аналогия для жюри:** «Как супервизор смены: 99% решений — автомат, 1% спорных — диспетчер с записью в журнал».

**Опционально на демо:** заранее подготовить 1 «грязный» объект с низким conf → показать вызов арбитра.

## Слайд 8 — Audit & метрики

`logs/events.jsonl` + **live-панель** на окне CV (`sim/metrics.py`):

| Метрика | Смысл для жюри |
|---------|----------------|
| Spawned / Removed | Бесконечный поток без утечки памяти |
| Scanned → Diverted | Полный цикл WCS |
| Divert accuracy % | В демо: совпадение рукава с demo-fallback; на проде — сверка с WMS |
| YOLO detections | Честное CV, не цвет движка |

## Q&A — заготовки

| Вопрос | Ответ |
|--------|-------|
| Почему симуляция, а не железо? | ТЗ допускает Digital Twin; CV и логика боевые |
| Почему не CoppeliaSim? | PyBullet = pip, один процесс, камера нативно |
| Зачем LLM? | Только спорные кейсы + audit; не в hot path |
| Как у Ozon на проде? | Штрихкод → WMS → рукав; CV — тип упаковки; WCS/ПЛК — тайминг |
| chute_a = тип A? | Нет. Рукав = направление. См. [BUSINESS_RULES.md](docs/BUSINESS_RULES.md) |
| Точность? | Метрики из events.jsonl + hold-out видео |
