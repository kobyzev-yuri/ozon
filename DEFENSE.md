## Питч на 3 минуты

**Рекомендация:** запустить симулятор заранее. На экране — тестовые объекты из набора организаторов.

### [0:00 – 0:45] Проблема и задача

«Перед основным сортировщиком нужен **предсортировочный ПАК**: отделить товары, которые можно пустить в основной контур, от негабарита и от позиций с «кругом в сечении», требующих доупаковки.

По ТЗ — три категории и три зоны: **B** (подходит), **C** (габарит), **D** (доупаковка). Ошибка здесь — заторы и ручной труд дальше по цепочке.»

### [0:45 – 1:45] Демонстрация

«На экране — PyBullet-симуляция с **реальными STL** из тестового набора (бутылка, короб, цилиндр…).

1. **Точка A** — подача на ленту.
2. **Classifier** (`PacClassifier`) — габариты 10×10×2…450×320×320 мм, затем критерий круга в сечении (порог 0,7).
3. **Маршрутизация** — `routes.yaml`: category → zone_b / zone_c / zone_d.
4. **Исполнительная часть** — cross-belt актуатор по ETA; маркеры B/C/D на сцене.

CV (YOLO + ByteTrack) — локализация; в симе ground truth категории из геометрии mesh.»

### [1:45 – 2:30] Метрики

| Метрика | Смысл |
|---------|--------|
| Category / zone accuracy | Совпадение с эталоном из `stl_analysis.json` |
| Throughput | Объектов в минуту |
| Divert accuracy | Попадание в нужную зону B/C/D |

### [2:30 – 3:00] Заключение

«Целостный контур: **классификация → управляющий сигнал → физическая маршрутизация**. Симуляция допустима по ТЗ; физический прототип — следующий этап. Готовы к вопросам.»

## Топ-5 каверзных вопросов жюри и ответы

> *Раздел ниже частично устарел (WMS-формулировки). При ответах опирайтесь на [task_3.md](task_3.md) и категории B/C/D.*

### 1. «Два объекта слиплись на ленте — как отсортируете?»

**Ответ:**

«Это **cluttering**. На проде решается **индукцией** — лента разгона создаёт зазор. В коде:

- `InductionFilter` — `min_bbox_gap_px`, `MIN_TRACK_LENGTH`: не сортируем «склеенные» треки.
- **YOLO + ByteTrack** — два bbox, два `track_id`, если визуально разделимы.
- **`CommandQueue`** — одна команда на `track_id`; коллизии команд на актуаторе исключены.

На демо при слипании можем показать событие `no_read` или отложенный scan — это честное поведение, не silent fail.»

**Модуль:** `field/induction.py`, `planning/command_queue.py`

---

### 2. «50 линий × GPU — как масштабировать?»

**Ответ:**

«Не нужен серверный GPU на каждую линию.

1. **YOLO11n / YOLOv8n** — лёгкие веса, в коде `config/pipeline.yaml`.
2. Экспорт **ONNX → TensorRT / OpenVINO** — инференс на edge: Jetson, Intel iGPU, Hailo.
3. CV на **30 FPS** достаточно для сортировки; физика ленты — в ПЛК, не в нейросети.
4. **WMS** (`routes.yaml`) — один на кластер; WCS — один контроллер на линию.

GPU на хакатоне — для разработки; прод — edge.»

---

### 3. «Товар развернулся не той стороной — ИИ не узнает?»

**Ответ:**

«Камера **строго сверху** (top-down) — как на реальных линиях Ozon ([кейс ОВХ](https://www.pvsm.ru/machine-learning/391187)).

При обучении:
- аугментации: rotate 90°/180°/270°, brightness, noise;
- при необходимости — **YOLO-seg** по контуру, не по одной грани.

В симуляторе сравним объекты со случайным `y_offset` — модель не привязана к «идеальной» позе.

Для пограничных случаев — **LLM Arbitrator** (`arbitrage/llm_arbitrator.py`): crop + metadata → зона + reasoning в `arbitrator.jsonl`.»

---

### 4. «Фиксированная задержка — лента проскользнет, пушер промахнётся?»

**Ответ:**

«В **нашем** коде нет `time.sleep()` и не опора на `time.time()` для актуатора.

- ETA: `distance = actuation_x − position_x`, шаги = `distance / belt_speed` → **`CommandQueue.schedule(execute_at_frame)`**.
- Трекинг (**ByteTrack**) обновляет позицию каждый CV-кадр — аналог энкодера.

**На проде** финальный триггер — **оптическая завеса** перед актуатором: WCS «вооружает» рукав, датчик даёт точный импульс. В симуляции это следующий шаг — `ACTUATION LINE` по world X или триггер при пересечении линии в кадре.

Проскальзывание компенсируется **пересчётом ETA** по velocity трека (уже заложено в `PositionTracker`).»

**Модуль:** `planning/timing_controller.py`, `sim/runner.py` (`WorldTiming`)

---

### 5. «Почему PyBullet, а не Factory I/O или ROS 2?»

**Ответ:**

| | PyBullet | Factory I/O | ROS 2 |
|--|----------|-------------|-------|
| Установка | `pip install` | лицензия + PLC | стек, DDS |
| Видео в YOLO | `getCameraImage` нативно | capture экрана | камера + bridge |
| Для хакатона MVP | **идеален** | если акцент Modbus | overkill |

«Мы — **CV-first** команда. PyBullet дал единый Python-контур: физика + камера + YOLO в одном процессе без сетевых задержек — **Digital Twin для проверки гипотез**.

Архитектура **модульная** (`FrameSource`, `MockWMS`, `SimActuator`), не один файл. На проде: RTSP вместо PyBullet, Modbus вместо `applyExternalForce` — **интерфейсы те же**. Factory I/O — опция, если жюри требует ПЛК-нарратив.»

## Дополнительные вопросы (коротко)

| Вопрос | Ответ в одну фразу |
|--------|-------------------|
| Почему не только штрихкод? | На проде главный — штрихкод WMS; CV — тип упаковки и fallback при No Read (`pyzbar` на SCAN LINE). |
| chute_a — это тип A? | Нет. Рукав = направление (WMS); тип box/sphere = YOLO. См. [BUSINESS_RULES.md](docs/BUSINESS_RULES.md). |
| Как докажете точность? | KPI на экране + `events.jsonl` + `metrics.export_snapshot()`. |
| Это не «подглядывание» в движок? | Детекция только с RGB кадра; класс для WMS после YOLO+match, не `getVisualShapeData`. |
| LLM не медленный? | Арбитр **не в hot path** — только low confidence, rate limit в config. |
| Чем отличаетесь от конкурентов? | Слои Ozon WMS/WCS/ПЛК + audit + опциональный LLM-диспетчер. |
| Что если система ошибётся? | [docs/ERROR_CASES.md](docs/ERROR_CASES.md) — 13 кейсов с диагностикой и fix. |

## Пояснительная записка (структура для сдачи)

### 1. Актуальность

Масштабирование ФЦ → заторы при No Read, нестандартная упаковка. Цель: **WMS-маршрут по штрихкоду** + CV для типа упаковки и подстраховки в реальном времени.

### 2. Архитектура

| Слой | Реализация |
|------|------------|
| Симуляция (Field) | PyBullet: лента, спавнер, актуатор |
| CV (ИИ) | YOLO11 + ByteTrack, top-down камера |
| WCS | ScanStation (+ блок коррекции), PositionTracker, CommandQueue |
| WMS | `RoutingTable` / `routes.yaml` |
| Коррекция маршрута | LLM Arbitrator на SCAN (опционально) |
| Коррекция исполнения | `fault_sim` в PyBullet (slip, miss пушера) |
| Observability | `events.jsonl`, KPI Dashboard |

### 3. KPI Dashboard

- **Throughput** — шт/мин (`throughput_per_min`)
- **AI Accuracy** — processed / (processed + ai_missed)
- **Счётчики** — `processed_boxes`, `processed_spheres`
- **AI Missed** — объект покинул ленту без скана/сортировки

### 4. Экономический эффект

- Камеры общего назначения vs дорогие ОВХ-рамки (на части сценариев).
- Меньше ручной выбраковки при повреждённой маркировке.
- **Инвариантный IT-код:** RTSP + Modbus вместо симулятора.

### 5. Перенос на реальный склад

```
get_camera_stream()  →  FrameSource (RTSP / Hikvision)
applyExternalForce   →  SimActuator → Modbus TCP → ПЛК
routes.yaml          →  API WMS Ozon
events.jsonl         →  WCS event bus
```
